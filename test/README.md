# 벽 자동 인식 회귀 가드

`wall-snapshots.json` = 픽스처 도면별 벽 인식 결과의 **좌표 단위 스냅샷**
(각 항목: "fromX,fromZ,toX,toZ,두께[,창N]" 정렬 목록).

## 프로토콜
- trace.js를 변경하면 픽스처 전체를 재실행해 이 스냅샷과 **좌표 단위로** 대조한다.
  개수 비교만으로는 재구성이 은폐된다(실사례: 밴드 룰 변경이 CAD 벽을 재구성했지만
  개수가 우연히 같아 가드를 통과했음).
- 차이가 나면: (1) 의도된 개선인지 판별하고 근거와 함께 스냅샷 갱신, (2) 아니면 롤백.
- 실전 지표는 별도: 합성 하드셋(ai-auto/도면샘플/실전풍_*) recall/precision + LH 배치 커버리지.

픽스처 원본: ai-auto/도면샘플/ (버전 관리 밖). widthCm은 스냅샷에 기록됨.

## v2(trace2.js) GT 랩 벤치 프로토콜 (2026-07-12 구축)

정답 있는 합성 하드셋에서 v1/v2를 같은 심판으로 채점한다.

1. GT 랩: `ai-auto/도면샘플/실전풍_생성.py` → `실전풍_아파트_850px.jpg` + `실전풍_아파트_GT.json`

   (세그먼트 vs 세그먼트, 1px=1cm 래스터, tol ±6cm. 보정 ±20% 실험은 --assumed-width로 재스케일)
3. 실행: `npm run dev` 후 headless Chrome에서
   `(await import('/src/lib/trace2.js')).detectWalls2('/bench.jpg', {x:0, z:0, widthCm})`

  잔존(문턱 235에서 존 밝은 픽셀 5%, 마루 텍스처 상한 222와 분리 불가) — 신호가
  물리적으로 부족. 벽 회수는 톤 분할(5c)이 담당, 창 승격은 포기.

v1 우위(48/59승). 단 **상보성 뚜렷**: v1 붕괴 시트(F1<0.5, 12장 — 벽 수백~2249개
junk 폭주 포함) 중 9장에서 v2 우위. v1 정상 시트(F1≥0.75, 39장)에서 v2 중앙값 0.700.

지지 — v1 junk 폭주 감지(벽 수/픽셀 비율) 시 v2로 폴백하는 안이 유력.

v2 남은 과제: ①CubiCasa 약점 개별 진단(recall 갭·폭 추정 강건화) ②combiner 설계
③cad 잔여 갭(0.738)=가구 외곽선 다중 홉 분지 분할 — v1 홈그라운드.

### recall 갭 진단 (2026-07-13, 4장 라벨맵 — 13278·13019·1025·11398)

v2 저 recall의 실체(빈도순):
1. **다중 도면 시트 최대군집 단독 선택** (1025·11398 실측): CubiCasa test엔 한 시트에
   2개 층/유닛이 흔한데 v2는 bestCluster 하나만 처리 → 나머지 층 전체 미검출.
   v1은 제한 없음. **v2 recall 최대 레버 = 멀티 클러스터 지원**(배너 군집과의 구분
   설계 필요 — 배너는 채색 솔리드, 도면 군집은 가는 암선 구조).
2. **타일 격자 욕실 바닥 → 닫힘이 통짜 구조 슬래브로 봉합** (13278·13019): 격자 셀이
   닫힘 직경보다 작으면 solid가 돼 분지 소멸·주변 벽 리지 전멸. 8차의 '두꺼운 암부
   재분류'는 raw 기준이라 mesh엔 무력(raw에선 가는 선). 처방 후보=닫힘 후 solid가 된
   대형 영역을 raw 밀도로 재검사.
3. **거실→외부 누수**: 창 새시가 흰 띠(잉크 없음)면 방이 외부 라벨로 새서 그 방의
   모든 벽이 외부-외부 쌍이 됨(기장 침실1-발코니와 동일 모드).
4. **폭 과대추정**: 마케팅 배너·다중 도면이 컨텐츠 폭을 부풀림(13278: 추정 2180 vs
   실제 ~800cm) → cm 문턱 전반이 2~3배 어긋남. autoCalibrateWidth의 구조정합 스캔이
   배너를 구조로 오인.
5. **점선 심볼·문호 파편화**: 점선 가구·시선 표시가 방을 색종이(confetti) 분지로
   조각 → 병합이 못 따라가고 가구 경계 리지 junk.

### v2 18차: 멀티 도면 범위·군집 (2026-07-13)

recall 갭 진단 1번(다중 도면 시트)의 처방. 진범은 군집 선택이 아니라 **3c 건물
범위의 단일 bbox** — 최대 성분에서 25cm 연쇄로 자란 박스 하나만 인정해서, 떨어진
두 번째 층/유닛의 방 전체가 '범위 밖 과반'으로 외부 흡수되고 있었다(1025·11398
라벨맵에서 두 번째 도면이 군집 단계에 도달조차 못 함을 실측).

바꾼 것 2가지:
1. **다중 범위**: 씨앗(잔여 중 nStruct 최대)→성장 박스 완성을 성분이 소진될 때까지
   반복. 첫 그룹은 기존 단일 범위와 동일. 게이트 = 그룹 구조 질량 ≥ 최대 그룹의
   20%(실측 분리: 진짜 2번째 층 83·94% vs CAD 캡션·LH 라벨띠 3% — 캡션 그룹을
   살리면 여백 치수띠 방·유령 벽이 되돌아온다). 흡수 판정은 인정된 박스 합집합.
2. **박스별 지배 군집 유지**: 방 라벨→박스 배정(픽셀 다수결)을 흡수 패스에서 얻고,
   군집→박스는 구성 방 면적 가중 다수결. 인정된 박스마다 최대 면적 군집 1개씩
   keep — 같은 박스의 비지배 군집(가구 그늘·배너 조각)은 기존처럼 버려진다.

결과: 1025 F1 0.382→**0.694**(v1 0.638 역전)·11398 0.485→**0.745**·13438 0.497→0.737·
1569 0.517→0.763·1869 0.713→0.793·5135 0.159→0.368·7790 0.060→0.163 등 —
**CubiCasa v2 평균 0.587→0.611(중앙값 0.640→0.664), 악화 시트 0**.
무회귀 확인: GT랩 3보정 f1 0.966/0.935/0.967 동일, 픽스처 12종 벽 수·F1 전부 동일
(cad 흡수 3건·기장 흡수 7건 유지 — 20% 게이트가 캡션·라벨띠 부활을 차단).
combiner 갱신 효과: mean 0.704→**0.708**(오라클 0.711), regret_max 0.150→**0.056**,
v2 픽 12건 중 10건 정확(5135가 v2 개선으로 오답→정답 전환).

### combiner (2026-07-13, `src/lib/combine.js` — 에디터 기본 경로)

**detectWallsAuto = 두 엔진 실행 후 상호 지지도로 선택.** 신호=v2 벽 표본점(30cm
간격)이 v1 검출 근방(tol 12cm+두께/2)에 있는 비율(support). 정상 도면은 두 엔진이
서로를 재발견(support 높음), v1 붕괴 시트는 v2 벽을 v1이 못 받침(support 낮음).
v1 junk 폭주는 아무 데나 벽이라 support가 높은데 그 경우 F1도 v1이 나아(junk가
GT를 넓게 덮음) 신호 방향이 일치 — 별도 junk 가드 불요.

검증(문턱 0.5):
- CubiCasa 60장: **med 0.783 / mean 0.704 / 정답 54/60 / regret 0.33** —
  오라클(mean 0.709)에 근접, always-v1(0.679)·always-v2(0.587) 모두 상회.
  v2 폴백 12건 중 9건 정확(3529 v1=0벽→0.449 구조 등), 오폴백 3건은 양 엔진 모두
  나쁜 시트라 손실 미미(최대 0.15). 문턱 0.3~0.55 평평(과적합 아님).
- 픽스처 9종+hard+LH 2종(홈그라운드): **전부 v1 유지**(support 0.80~1.00) — 오폴백 0.
- JS 실구현 vs 파이썬 시뮬레이션 픽 불일치 0/60.
- 한계(수용): hard는 v2(0.966)가 v1(0.884)보다 낫지만 support 높아 v1 선택 —
  combiner의 역할은 붕괴 구조이지 미세 최적화가 아님.

표기. 재검증 스크립트(전부 `ai-auto/도면샘플/CubiCasa5K/`에 보관): `combine_bench.mjs`
